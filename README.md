## What it does

- **Library** ‚ÄĒ `compress(messages)` in Python or TypeScript, inline in any app
- **Proxy** ‚ÄĒ `headroom proxy --port 8787`, zero code changes, any language
- **Agent wrap** ‚ÄĒ `headroom wrap claude|codex|cursor|aider|copilot` in one command
- **MCP server** ‚ÄĒ `headroom_compress`, `headroom_retrieve`, `headroom_stats` for any MCP client
- **Cross-agent memory** ‚ÄĒ shared store across Claude, Codex, Gemini, auto-dedup
- **`headroom learn`** ‚ÄĒ mines failed sessions, writes corrections to `CLAUDE.md` / `AGENTS.md`
- **Reversible (CCR)** ‚ÄĒ originals never deleted; LLM retrieves on demand

## When to use ¬∑ When to skip

**Great fit if you‚Ä¶**
- run AI coding agents daily and want savings without changing your code
- work across multiple agents and want shared memory
- need reversible compression ‚ÄĒ originals always retrievable via CCR

**Skip it if you‚Ä¶**
- only use a single provider's native compaction and don't need cross-agent memory
- work in a sandboxed environment where local processes can't run

<details>
<summary><b>What's inside</b></summary>

- **SmartCrusher** ‚ÄĒ universal JSON: arrays of dicts, nested objects, mixed types.
- **CodeCompressor** ‚ÄĒ AST-aware for Python, JS, Go, Rust, Java, C++.
- **Kompress-base** ‚ÄĒ our HuggingFace model, trained on agentic traces.
- **Image compression** ‚ÄĒ 40‚Äď90% reduction via trained ML router.
- **CacheAligner** ‚ÄĒ stabilizes prefixes so Anthropic/OpenAI KV caches actually hit.
- **IntelligentContext** ‚ÄĒ score-based context fitting with learned importance.
- **CCR** ‚ÄĒ reversible compression; LLM retrieves originals on demand.
- **Cross-agent memory** ‚ÄĒ shared store, agent provenance, auto-dedup.
- **SharedContext** ‚ÄĒ compressed context passing across multi-agent workflows.
- **`headroom learn`** ‚ÄĒ plugin-based failure mining for Claude, Codex, Gemini.

</details>

<details>
<summary><b>Pipeline internals</b></summary>

Headroom exposes one stable request lifecycle across `compress()`, the SDK, and the proxy:

`Setup` ‚Üí `Pre-Start` ‚Üí `Post-Start` ‚Üí `Input Received` ‚Üí `Input Cached` ‚Üí `Input Routed` ‚Üí `Input Compressed` ‚Üí `Input Remembered` ‚Üí `Pre-Send` ‚Üí `Post-Send` ‚Üí `Response Received`

- **Transforms** do the work: CacheAligner, ContentRouter, SmartCrusher, CodeCompressor, Kompress-base, IntelligentContext / RollingWindow.
- **Pipeline extensions** observe or customize lifecycle stages via `on_pipeline_event(...)`.
- **Compression hooks** sit alongside the canonical lifecycle as an additional extension seam.
- **Proxy extensions** remain the server/app integration seam for ASGI middleware, routes, and startup policy.

Provider and tool-specific behavior lives under `headroom/providers/` so core orchestration stays focused on lifecycle, sequencing, and policy.

- **CLI/tool slices**: `headroom/providers/claude`, `copilot`, `codex`, `openclaw`
- **Provider runtime slices**: `headroom/providers/claude`, `gemini`, plus shared backend/runtime dispatch in `headroom/providers/registry.py`
- **Core files stay orchestration-first**: `wrap.py`, `client.py`, `cli/proxy.py`, and `proxy/server.py` delegate provider-specific env shaping, API target normalization, backend selection, and transport dispatch.

</details>
